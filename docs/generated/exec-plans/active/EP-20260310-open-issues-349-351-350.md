# ExecPlan: Open Issue Flow for #349, #351, #350

## Goal

Open issue のうち current `main` に対して小〜中規模で着手可能な backend issue `#349`, `#351`, `#350` を優先順に処理し、focused validation と `jj` snapshot を経て PR 作成まで完了する。

## Why these issues first

- `#349` は未使用コード除去で影響範囲が狭い
- `#351` は bind 型の不整合解消で挙動変更がない
- `#350` は repository 境界を戻す中規模 refactor だが、`#349` と同じ refresh rotation 周辺の整理として続けて処理できる

## Steps

- [completed] issue `#349` の未使用関数と抑制コードを除去し、関連テストを確認した
- [completed] issue `#351` の `reset_user_mfa` bind 一貫性を整えた
- [completed] issue `#350` の refresh rotation SQL を repository layer へ移し、handler の raw SQL を除去した
- [in_progress] focused validation 後 `jj commit` を作成する
- [pending] 変更を push して GitHub PR を作成する
